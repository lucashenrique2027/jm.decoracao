BRIEFING COMPLETO DO PROJETO

TCC — Plataforma de Vendas para Ateliê Artesanal

Vidros, Potes, Copos e Garrafas com Arte Estilizada

Atibaia — SP • Stack: Vite + React + Supabase + Cloudflare

1. Contexto do Negócio

Este documento é o briefing de contexto completo para o desenvolvimento
do TCC. Qualquer instância de IA ou desenvolvedor que o leia deve ser
capaz de operar com o mesmo nível de entendimento construído ao longo
das sessões de levantamento de requisitos.

1.1 Quem é o cliente

-   Dono de um ateliê artesanal em Atibaia — SP, operado há décadas

-   Produz manualmente e estoca: potes, copos, garrafas e vidros com
    desenhos estilizados

-   Opera com uma equipe pequena de parceiros e colaboradores

-   Possui equipe própria de entrega local

-   Vende centenas de produtos por dia — volume real, não hobbyista

1.2 Como vende hoje

-   100% por WhatsApp: exibe fotos da galeria, cliente escolhe, paga,
    manda localização

-   Clientes chegam por indicação boca a boca e loja física

-   Nenhuma presença digital estruturada — tudo manual e humano

-   O próprio dono é o ponto central de atendimento, catálogo e caixa ao
    mesmo tempo

1.3 O problema real

O dono não tem um problema de vendas — ele já vende. O problema é que
ele é o único ponto de contato para tudo: exibir produtos, responder
perguntas, receber pedidos, combinar entrega e confirmar endereço. O
site precisa absorver as partes repetitivas e operacionais desse fluxo,
liberando o humano para o que só ele faz: produzir e relacionar.

2. Definição do Produto

2.1 O que o site É

Uma plataforma de catálogo e pedidos para ateliê artesanal de médio
volume, com foco em apresentação visual de qualidade, autonomia do dono
para gestão do catálogo e redução do atendimento manual repetitivo.

2.2 O que o site NÃO É

-   Não é um Shopify — sem gateway de pagamento online

-   Não é marketplace — sem múltiplos vendedores

-   Não é sistema de gestão de estoque em tempo real complexo

-   Não é plataforma de escala nacional — atende apenas Atibaia e
    cidades vizinhas

2.3 Fluxo central do produto

Cliente acessa o site (filtrado por geolocalização — Cloudflare)
↓
Navega pelo catálogo (modelos fixos + peças únicas)
↓
Chatbot responde dúvidas e sugere produtos por linguagem natural
↓
Cliente se cadastra com endereço completo + CEP
↓
Sistema valida CEP na zona de entrega (hard block se fora)
↓
Cliente monta pedido no site
↓
Pagamento combinado fora (Pix / presencial)
↓
Dono recebe pedido no painel com endereço, itens e observações
↓
Dono confirma, agenda entrega, emite nota fiscal

3. Requisitos Definidos

3.1 Requisitos funcionais confirmados pelo cliente

-   Tela de login para o dono — dados usados para emissão de nota fiscal

-   Painel de controle para gerenciar estoque: adicionar, remover e
    editar produtos

-   Catálogo público com produtos (modelos fixos + peças únicas)

-   Endereço do cliente capturado no momento do pedido

-   Restrição de entrega por zona geográfica — bloqueia CEP fora da área

-   Campo de observação de entrega (referência, horário, portão)

3.2 Requisitos técnicos definidos

-   Stack: Vite + React (frontend já iniciado com cards visuais básicos)

-   Backend/DB/Auth/Storage: Supabase

-   Geobloqueio camada 1: Cloudflare (regras por cidade — elimina
    tráfego que não converte)

-   Deploy: Vercel ou Netlify (a definir)

-   Imagens dos produtos: Supabase Storage (decisão fechada — unifica a
    plataforma)

-   Autenticação: Supabase Auth com sistema de roles

4. Arquitetura de Restrição Geográfica

O cliente atende apenas Atibaia e cidades vizinhas com sua equipe de
entrega própria. O site não deve criar compromissos de entrega que ele
não consiga cumprir. A solução é em camadas:

  ----------- ------------------ ------------------ ---------------------

                                                    aceita?

              geobloqueio por    que não converte   mas não é barreira de
              cidade                                segurança

  2 —         Validação de CEP   Impedir conta fora Baixa — CEP pode ser
  Cadastro    no cadastro do     da zona            digitado errado
              cliente                               

  3 — Pedido  Revalidação de CEP Garantir endereço  Mínima
              no momento do      real na entrega    
              pedido                                

  4 — Humana  Dono confirma      Última barreira —  Nenhuma
              pedido no painel   decisão humana

Importante: geobloqueio por IP não é solução de segurança — é filtro de
tráfego. Isso é tecnicamente honesto e deve ser documentado assim no
TCC.

9. Princípios que Guiam o Produto

Estes princípios devem ser usados para avaliar qualquer decisão de
produto ou técnica durante o desenvolvimento:

1. O software absorve o repetitivo, o humano fica com o relacional.

O dono já resolve tudo — o sistema poupa trabalho, não substitui
julgamento.

2. Não criar compromisso que o negócio não consegue cumprir.

Validação de CEP é proteção ao dono, não burocracia para o cliente.

3. Simplicidade operacional para o admin.

O dono não é técnico. Qualquer feature do painel precisa ser usável sem
treinamento.

4. A ausência de carrinho convencional é decisão, não limitação.

Produtos artesanais têm variação e personalização. A conversa com o
cliente é parte do valor.

5. Cada camada de restrição geográfica tem um objetivo diferente.

Cloudflare filtra tráfego. CEP bloqueia pedido inentregável. Humano
decide o edge case.
